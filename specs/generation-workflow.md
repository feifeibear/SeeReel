# Generation Workflow

Status: active
Owner: SeeReel
Last Reviewed: 2026-06-07

## Purpose

Define how SeeReel creates, edits, reviews, retries, and stitches generated media while keeping user-visible canvas state as the source of truth.

## Scope

- Script, image, storyboard, video, review, retry, and stitch workflows.
- Prompt editing before and during generation.
- Seedance, Seedream, Seed text generation, manual VLM review, and final stitching state.
- API and UI behavior for generation nodes.
- CLI-created sessions and browser handoff behavior for human takeover.
- CLI cloud-only production runs where user-provided local素材 may be uploaded once, while generated intermediates are created and stored by SeeReel server tasks.
- Cross-agent skill packaging and routing for Codex, Claude Code, Cursor, and generic Agent Skills runtimes.
- Agent skill orchestration that routes initial idea -> script -> casting/assets -> cinematography -> final canvas review -> approved generation.
- Script-development workflows that collect character, plot, and historical background material before first draft, then revise through review passes before locking `StoryPlan`.
- Casting and production-design workflows that create/import/review image references for characters, scenes, props, style, moodboards, and other visual material while keeping `assetIds` auditable.
- Cinematography workflows that turn locked `StoryPlan` plus approved assets into storyboards, shot nodes, camera grammar, continuity wiring, and Seedance-ready `rawPrompt`/`prompt`.
- Final canvas-review workflows that compare all visible canvas nodes, prompts, edges, assets, storyboards, and continuity fields back to the initial idea and locked StoryPlan before video generation.
- Session style-consistency workflows that keep recurring characters, scenes, props, storyboards, and shot prompts in one visual family before render.
- Narrative orientation workflows that make character identity, era/background, scene arena, world rules, relationships, and stakes audience-facing before render.

## Non-Goals

- This spec does not define provider pricing.
- This spec does not require paid generation for ordinary local smoke tests.
- This spec does not define every provider-specific parameter.

## User Stories

- As a creator, I can edit prompts while shaping a workflow and see the saved value in the node.
- As a creator, I understand whether the current running provider request used the old prompt or the new prompt.
- As an operator, I can audit which prompt, references, assets, and review settings produced each render.
- As a human reviewer, I can claim a CLI-created workflow in my browser and continue editing it without sharing the CLI cookie identity.
- As a creator, I can choose among BytePlus standard API, Volcengine CN standard API, and Agent Plan credentials for Seedream image generation and Seedance video generation.
- As an AI agent, I can discover the right SeeReel skill for the user's intent and operate the same visible app state across Codex, Claude Code, Cursor, and CLI package installs.
- As a creator, I can export a complete session package from the UI and import it on another machine as my own editable copy.

## Product Rules

- Canvas state and SeeReel APIs are the source of truth for prompts, references, renders, review settings, stitch order, and final video URLs.
- Manual filesystem recovery is allowed only as a recovery step, and recovered media must be written back into visible app state.
- In CLI cloud-only mode, local files are allowed only as user input references. Characters, scenes, storyboards, shot videos, stitch outputs, and final videos must be produced through SeeReel APIs/server jobs, persisted in canvas state, and not fabricated from local scratch media.
- Generated video workflows do not emit subtitle files or burn subtitles by default. The post-stitch audio-track node is the only allowed subtitle surface: it may either add no subtitles or burn the narration script into the narrated mp4 when the user explicitly selects that option and position.
- Audio-track nodes are independent optional post-production nodes. They can add either OpenSpeech narration or Volcengine `doubao-music` BGM/song audio after a successful stitch. Music generation must use Volcengine OpenAPI AK/SK signing against `open.volcengineapi.com` (`Service=imagination`); Agent Plan keys are not a valid music fallback.
- The Full video Inspector also exposes a post-production package panel for publishing polish: title card, optional cover image, manually entered or automatically transcribed subtitles/SRT, and optional Volcengine TTS voiceover or Volcengine music. This produces a separate `postProductionVideoUrl` MP4 and never overwrites `finalVideoUrl` or silently creates canvas edges. The package signature must include the source final-video signature plus title, cover, subtitle, audio, and volume options, so any source-video or packaging change requires an explicit regenerate and cannot reuse stale media. Automatic subtitles must prefer the existing post-production voiceover script when present, generating SRT directly from that text without ASR; otherwise they default to Volcengine/Doubao Speech ASR (`POST_PRODUCTION_ASR_PROVIDER=volc`) against the current full-video audio, convert returned utterance timestamps into short readable SRT cues, and fill the editable subtitle box before rendering. Subtitle-only burns must default to a low-memory ffmpeg/ASS renderer that preserves the source video resolution and copies existing audio; long SRT cue text must be split into multiple timed cues and each ASS dialogue must insert explicit safe-width line breaks so subtitles remain inside the video frame. Title cards, cover images, and other visual packaging stay on HyperFrames so those visuals are not silently dropped. HyperFrames `transcribe --json` remains an explicit local fallback only when `POST_PRODUCTION_ASR_PROVIDER=hyperframes`.
- Audio-track nodes can also run 人声/背景分离 against the current Full video output. The result must persist two playable audio stems on the session: human voice and background. Deployments may provide a real separation backend through `AUDIO_SEPARATION_COMMAND`; local/dev fallback may use ffmpeg center-vocal extraction so the canvas workflow remains reviewable without an external model.
- Like LibTV, SeeReel exposes audio as a visible canvas node. The right-click menu can create an audio-track node with shortcut `A`; it binds to a generated Stitch node output and opens the post-production Inspector for narration or music.
- 声音节点 is a separate reusable audio identity node, not the post-stitch audio-track node. The right-click menu can create a Voice node with shortcut `V`; its Inspector presents a visible preset-card picker for common tones such as 少女音色、东北丫头、东北老铁、青年男声, persists `voicePresetId`, stores the voice prompt, optional advanced provider `voiceId`, preview text, and a playable TTS preview audio file. Selecting a preset must write the preset's provider voice id, while an explicit advanced `voiceId` still wins. Voice prompts such as age, gender, language, and accent/dialect must be resolved to the closest supported provider voice id when the user has not supplied an explicit `voiceId`; explicit `voiceId` still wins. TTS preview must choose a `resourceId` compatible with the selected provider speaker and retry with the inferred compatible resource when Volcengine returns a resource/speaker mismatch. Audio-track voiceover mode can select a Voice node so generated narration keeps the same configured or inferred voice id and leaves `narrationVoiceAssetId` auditable on the session.
- Final-video stitching is controlled from a stitch button/panel. The visible Stitch node can remain on the canvas as a final-video artifact/entry point, but ordinary stitching must not require every video node to draw a canvas edge into it.
- The Stitch Inspector opens a playlist UI over the canvas video nodes. It defaults to video creation order, lets users drag available video nodes into the playlist, drag playlist items to reorder them, remove videos from the playlist, and persists the chosen order to `stitchShotIds` or a `StitchJob.shotIds` playlist before running stitch.
- Final-video outputs keep their last playable mp4 when source videos or playlist membership changes, but `/api/state` must expose the current stitch input signature and mark the output stale when the saved final signature no longer matches. SeeReel must not auto-restitch or spend generation/ffmpeg work silently; the user explicitly clicks "重新拼接" to sync the full video to the latest playlist. If the playlist is still default, newly ready video nodes enter the current input signature by creation order. If the playlist has been customized, newly ready video nodes remain outside the playlist and surface as "新视频未加入" until the user adds them.
- The Full video playlist must mark each selected video whose current render was generated after the current final video with an inline "已更新" / "Updated" badge, so users can see which playlist items make the final video stale before clicking "重新拼接".
- Shot-to-Shot video-node wiring must be allowed before the upstream shot has rendered. The edge persists `referenceVideoFromShotId` immediately so users can plan the chain first; generation only sends Seedance `reference_video` once the upstream shot has a resolvable video URL. Users must be able to drag between video nodes without knowing which tiny handle is the strict source or target.
- Rendered video nodes and derived tail-clip video nodes must play directly on the canvas with native controls for play, pause, progress, and seek. Interacting with the video controls must not drag, pan, or select the canvas node; no transparent drag layer may cover the player controls, and the node chrome outside the player remains the drag/move surface.
- The canvas right-click create menu must be able to create a visible storyboard node. Because storyboard nodes are shot-scoped, this action creates a new shot and enables its sub-storyboard state without submitting paid generation.
- The canvas right-click create menu must stay usable near viewport edges: bottom/right clicks flip the menu into view, and small viewports make the node list scroll inside the menu instead of clipping options off-screen.
- Right-click-created nodes must appear exactly at the clicked canvas position after React Flow projection. SeeReel must not apply a fixed center/offset shift after the user chooses a node type.
- Manual canvas node placement is part of session state. After a user drags nodes into a custom layout, refreshes and snapshot polling must preserve the saved positions instead of returning to the automatic column layout.
- The canvas uses a simplified LibTV-like visual node model: character references, scene references, props, style references, moodboards, and uploaded stills all appear to users as `Image` nodes. Legacy `Asset.type` values may remain for prompt composition and backward compatibility, but the canvas creation menu must not split image nodes into character/scene/style node kinds.
- Image nodes may reference each other for future image generation and image editing. Canvas edges between image nodes must persist to the target asset's `referenceAssetIds`; the next Seedream generation of the target image must use only the connected image references and the latest source asset fields.
- Image nodes must expose both a left target handle and a right source handle in the canvas so users can manually wire image-to-image editing chains.
- Image model-version selection belongs in the Image node Inspector, not on the canvas node. The Image Inspector must expose Seedream 5.0 Lite as a selectable image model without labeling it Agent Plan-only; BP/CN standard API and Agent Plan routes may all use it. Changing the Image Inspector model or resolution must update the visible control immediately, persist to the asset, and the next image generation must use that just-selected local value even before a full state refresh returns. API callers may pass either the canonical `seedream-5-lite` or alias `seedream-5.0-lite`. Video model-version selection belongs in the Video/Shot node Inspector, not in hidden prompt composition or canvas prompt tooling.
- Moodboards are image nodes tagged `moodboard` + `style-reference`. They only influence Seedream/Seedance when explicitly wired to an image, storyboard, or shot. Moodboards constrain visual tone, palette, texture, era, and lighting, but must not override character identity, scene geography, first-frame anchors, reference videos, or storyboard action.
- Dropping or picking a local image must create a visible pending Image node immediately, with a local preview and the user's intended canvas placement, before the server upload finishes. Once upload succeeds, the pending node must be replaced by the persisted cloud asset without losing the placement; if upload fails, the pending node must be removed and the user must see the upload error.
- `@` mention suggestions and generation-time reference resolution must be graph-wired only. A session-visible asset that is not connected to the target shot/storyboard/asset must not appear in the `@` list and must not be pulled into Seedream/Seedance merely because stale prompt text still contains its old name. Image nodes must visibly show their wired, `@`-referenceable upstream images on the canvas and in the Inspector prompt editor.
- When an Image node generation submits wired references to Seedream, the submitted prompt must rewrite prompt-mentioned attached images from `@XXX` to `参考图 N` in submitted-reference order, with no trailing reference-binding section. This is structural media binding, not AI prompt rewriting: `参考图 1` refers to the first image in the submitted `image` array, `参考图 2` to the second, and so on.
- When a Shot/Video node generation submits `@XXX` media references to Seedance, the submitted text must follow the BytePlus Playground reference-token convention: prompt-mentioned attached images are rewritten from `@XXX` to `【Pictures N】`, videos to `【Video N】`, and audio to `【Audio N】`, while the request body submits the matching ordered `reference_image`, `reference_video`, or `reference_audio` content item. The prompt must append only compact reference-binding metadata for `@` handles that were both present in the prompt and actually submitted as media. Image references map to `参考图 N / Pictures N / reference_image N`; video references map to `参考视频 N / Video N / reference_video N`; audio references map to `参考音频 N / Audio N / reference_audio N`. Plain prose names without `@`, unconnected stale `@` text, or references filtered out before payload submission must not create binding metadata.
- Seedream reference-image payloads must follow the BytePlus image generation API shape: pass reference images through the `image` field as an ordered array, cap the submitted list at the documented 14-image maximum, explicitly set `sequential_image_generation: "disabled"` for single-image output, keep `response_format: "url"` and `watermark: false`, and only set `output_format: "png"` for model variants that support custom output format such as Seedream 5.0 Lite.
- Script and storyboard planning must receive only assets owned by the current session, assets owned by the current session's shots, or global assets explicitly wired to the current session's shots. Local fallback planning must not auto-select the first assets from the global library when a new session has no relevant references.
- Reference resolution must store relationships by asset id and read the current asset record at generation time. If a connected asset's prompt, description, name, media URL, or TOS URL changes, dependent shot/storyboard/asset generation must use the latest asset fields without requiring users to reconnect the edge.
- Image asset previews must be derived from the current asset media fields. When a generated/uploaded image updates `mediaUrl` or `imageUrl`, any implicit `thumbnailUrl` must follow that current image unless the route explicitly writes a new thumbnail for the same image. Stale thumbnails from previous renders or other nodes must not override the canvas node or Inspector current-image preview.
- Image asset Seedream generation defaults to 2K resolution. The Inspector must expose a per-image-node resolution selector with 2K and 4K options, persist the choice on the asset, and submit that choice with the next Seedream generation request.
- Seedance reference inputs must be public or signed `http(s)` URLs, not local `/media/...` preview paths.
- Canvas previews for uploaded/generated images and videos must remain valid independently of expiring Seedream/Seedance/TOS presigned URLs. Still images should persist a lightweight local `/media/...` preview for canvas and Inspector display while retaining the original uploaded/generated image URL for generation references, download, and TOS publishing. Generated shot/final video canvas playback should prefer the local cached `/media/...` artifact before any remote playback/download URL.
- Tail-frame assets used as a next-shot first-frame anchor must be extracted from the strict final decoded frame of the source video, not from an approximate near-end timestamp.
- Video nodes must expose three derived-node actions: first frame, tail frame, and final 2-second tail video. Clicking any of these actions creates a standalone canvas node for the derived asset without automatically drawing a source edge from the video node; users may freely move those derived nodes and wire them later when they want continuity/reference behavior.
- A rendered shot node must be able to create a visible "tail clip" node from its final 2 seconds. The created asset is session-scoped, tagged as `reference-video` + `tail-clip` + `source-shot:<shotId>`, keeps a local preview, publishes to TOS when configured, and can be connected to the next shot as that shot's explicit reference video.
- Previous-shot continuity clips must be real trimmed tail clips, not merely metadata labels on a full previous shot. Users create them as visible tail-clip video nodes from the source shot, then wire or bind that visible node as the next shot's `reference_video`. A full previous-shot reference is only an acceptable fallback when tail extraction or publish fails.
- The video-node Inspector must not expose a hidden "previous tail clip" / "参考上一镜尾段" shortcut. Tail continuity belongs on the canvas as a visible tail-clip node and explicit reference-video relationship.
- If a provider request has not been submitted, prompt edits must affect the next submission.
- If a provider request has already been submitted, UI must make clear whether the current render uses the old prompt and whether retry/regeneration will use the new prompt.
- Video/Shot prompt edits auto-save from the Inspector without an explicit Save button. Generation must flush any pending auto-save before submitting the provider request so the user's latest visible prompt is the submitted prompt.
- While a shot is still generating, transient poll failures such as bare 500/502/503/504 gateway responses or network disconnects must not immediately show a global error. The UI should keep polling and surface the failure only after repeated consecutive failures for the same shot; provider terminal errors and non-transient configuration errors should remain visible immediately.
- Review controls must be interactive and must not be hard-coded differently between local and production.
- CLI and browser identities are isolated by `seereel_user_id` cookies on online deployments. A raw online CLI `webUrl` must not be described as browser-visible handoff; agents should return a one-time `handoffUrl` when a human needs to claim and edit the workflow.
- `localhost`, `127.0.0.1`, and `::1` CLI workflows should return the explicit session `webUrl` directly and must not create an encrypted handoff token for ordinary local review. Local requests are a shared review workbench: sessions created by the CLI under one cookie identity must automatically appear in browser `/api/state`, the session list, and `/canvas/:sessionId` for other local cookie identities.
- Handoff links for non-local deployments must be same-origin API links under `/api/handoff/:token`, unguessable, time-limited, one-time use, and transfer the session owner to the current browser identity before redirecting to the session workspace.
- Handoff must not make sessions public or weaken owner checks on non-local deployments; users without the original owner cookie or an unclaimed handoff token must keep receiving not-found responses for isolated online sessions.
- A CLI cloud-only run that renders video must be able to upload the user reference, generate server-side storyboard assets, render shots, stitch the final video, then download only the final cloud artifact to the user computer.
- 30s/60s one-shot requests must be planned as visible Seedance-sized shot chains with an explicit frame bridge plan. The director skill must teach agents to compute shot count with `Math.ceil(totalDurationSec / 15)`, design a route bible and rhythm map, choose first-frame/tailframe/reference-video modes intentionally, and keep all intermediate nodes visible in the cloud session. 30min requests are treated as an edge case, not the default workflow center.
- Narrative and dialogue workflows should prefer full 15-second Seedance shots. Agents should pack multiple related story beats into one 15s shot using internal motion beats when character, location, lighting, and emotional continuity are shared. Agents should not create one shorter video per beat unless a hard cut, location jump, time jump, or continuity reset is dramatically necessary; in-shot consistency is usually stronger than cross-shot consistency.
- Manually created video/Shot nodes default to 15 seconds in both the client and `appendShot` API. Existing shots keep their saved duration, and explicit shorter durations remain valid when the user chooses them.
- Narrative shots that share location, time of day, lighting, characters, or camera direction should not be treated as independent just because their story beats are different. Generate them serially and use visible tail-clip continuity nodes unless the cut is meant to be visibly hard. Use strict tailframe-to-first-frame anchoring when the next shot must begin on the exact final composition of the previous shot.
- Narrative shorts must have a saved story spine before render: title promise, protagonist, want/fear, antagonist or social pressure, escalation, reversal, and payoff.
- When a user gives a rough script idea or inspiration, the agent must research relevant characters, plot mechanisms, and historical background before writing the first complete draft. Research can use web search, local memory, user-provided references, or reliable primary/secondary sources, and the useful findings must be summarized in visible `StoryPlan` state.
- Research notes must explain how sources affect character assets, scene assets, beats, shot prompts, or anachronism constraints. Research must not remain only in private agent scratch notes.
- After the first script draft, `seereel-script-chat` must run script review iterations and patch the `StoryPlan` until the reviewer is satisfied or a bounded review loop reaches unresolved blockers. New script ideas require at least two review passes before `StoryPlan` lock.
- Unless the user explicitly asks for `交互模式`, discussion, or questions, the agent should work autonomously through research, drafting, review, revision, and canvas creation. Missing preferences should be inferred conservatively unless they are hard blockers.
- Every generated narrative shot must have a scene objective and must change story state. For 60s/4-shot satire or comedy, the director skill should teach a setup -> trap/pitch -> escalation -> reversal/payoff ladder unless the user specifies another structure.
- Dialogue-driven prompts must keep dialogue in `StoryPlan` beats, shot scripts, and prompt intent. They must describe dialogue as naturally performed action and must not depend on subtitles, readable signs, or text overlays to carry plot.
- Story-critical prompt information must be audience-facing. Agents must turn facts required to understand the plot, joke, scam, relationship, time jump, threat, or reversal into `audienceDelivery` entries delivered through voiceover or character dialogue, with visible action/reaction backup. Private prompt-only lore, subtitles, readable signs, UI text, captions, and text overlays do not count unless the user explicitly asks for readable text.
- Narrative shorts must include an opening orientation packet before render. By the first 10 seconds, a cold viewer should be able to answer who the protagonist is, where the story happens, when or what era/rule governs it, what relationship drives the scene, and why the moment matters. Every main character's first appearance must include `identityDelivery` through action, dialogue, voiceover, prop behavior, or a motivated establishing/master shot. The agent must not assume the viewer knows product names such as Codex, asset titles, prompt lore, or prior chat context.
- Dialogue-driven workflows must choose one spoken dialogue language for the session, inferred from the user's request or `Session.language`, and keep every quoted spoken line in that language across `StoryPlan`, shot scripts, and provider prompts. Technical prompt prose may be English, but quoted dialogue must not mix Chinese and English unless the user explicitly asks for multilingual dialogue.
- Image and video nodes must submit the user's prompt text verbatim. SeeReel must not AI-expand, auto-compose, append repair blocks, add language-lock text, add negative-prompt boilerplate, or otherwise modify prompt wording before provider submission. References, duration, ratio, first/last-frame mode, and media inputs may still be carried as structured provider fields, but the text prompt belongs to the user.
- VLM review is feedback-only and must be manually invoked. Image, storyboard, shot, and final-video generation must not automatically run VLM review, must not automatically retry because of VLM results, and must not block prompt editing or generation on VLM errors. Review failures may be shown in the UI and audit state, but automatic prompt repair routes, prompt rewriters, and repair buttons must not write changes back into image, storyboard, shot, or final-video prompts.
- Agent-written Seedance prompts should still be clear before paid render: shot identity, reference roles, entry frame, explicit shot size/framing, chronological `0-4s / 4-9s / 9-13s / 13-15s` motion beats, dialogue/sound, camera/style, exit frame, and negative constraints. Those words must be authored by the agent or user directly in the visible prompt field, not inserted by hidden runtime composition.
- Agent-written narrative prompts must include cinematic camera grammar before paid render: camera motivation, coverage ladder, axis/screen direction, blocking, attention target, and cut bridge. Camera movement must have a subject, speed, endpoint, and story reason; otherwise the agent should choose an intentional lock-off. Adjacent narrative shots should use continuity editing ideas such as 180-degree axis consistency, eyeline match, match on action, reaction shots, inserts, cutaways, sound bridges, previous-tail clips, or tailframes so the stitched film does not feel like unrelated generated clips.
- Reference-video continuity should use the shortest useful clip, usually 2-4 seconds around the needed motion, camera move, or handoff. Agents must not send an entire previous 15-second shot when only the final camera/motion cue is needed.
- Retry workflows should change one variable at a time and record the change in visible session notes or prompt metadata: prompt wording, reference set, trimmed reference duration, or generation parameters. Agents must avoid changing prompt, references, duration, ratio, and mode all in one retry when diagnosing a failed shot.
- Generated video prompts must default to natural diegetic sound only: spoken dialogue, room tone, footsteps, props, machinery, street ambience, wind, crowd murmur, breathing, and other in-world sounds. Agents must not add per-shot background music, BGM, score, soundtrack, stingers, or music cues by default because separately generated music does not stitch continuously. If the user explicitly requests music, it must be defined as one continuous session-level music bed rather than different music per shot.
- Post-stitch narration may use optional leading timecodes such as `[00:15]` or `[01:00]` on narration lines. Timecodes are timing metadata only and must not be spoken or rendered as subtitles. Agents should use timed narration lines when the voiceover must match specific stitched shot windows or story beats.
- `seereel-casting-assets` must establish a session style bible before recurring asset generation and reuse it in character, scene, prop, storyboard, and shot prompts. VLM style-mismatch reasons such as cartoon, anime, 3D render, illustration, plastic skin, toy proportions, or wrong-era look are blocking for recurring assets even when the numeric review score is high.
- `seereel-casting-assets` should extract every on-screen speaking or featured character as a session-scoped character asset whenever possible, not only the protagonist or originally named cast. Recurring, cross-shot, named, speaking, or featured characters need an explicit continuity plan; generated/imported visual assets are preferred, but prompt-only, storyboard-only, or rendered-frame continuity can be valid when recorded deliberately.
- Before render, recurring or featured characters need an explicit continuity plan. Character assets, rendered-frame anchors, storyboard references, or prompt-only continuity can all be valid when chosen deliberately; stale or accidental references should be removed before paid render.
- Seedance generation must not automatically attach the whole session cast to every shot. Session StoryPlan cast assets may be auto-added only when the shot prompt explicitly `@mentions` that character; manually wired `shot.assetIds` remain valid, but stale/unrelated cast must be removed before paid render.
- Once a rendered shot establishes an accepted recurring character face, later continuity repairs should prefer rendered-frame identity anchors extracted from that accepted video over the original lookbook. The extracted frames must be uploaded or published as visible session assets before reuse so the identity chain remains auditable.
- When a recurring asset is repaired or regenerated for style consistency, `seereel-casting-assets` must report dependent shots and `seereel-cinematography` must patch those shot prompts with a compact style guard before regenerating those shots.
- The staged review workflow must stop before video generation. The canvas must contain a saved locked `StoryPlan`, approved recurring character assets, approved recurring scene assets, shot nodes, per-shot prompts, storyboard prompts, and explicit `assetIds` links showing which character/scene/storyboard references each shot needs.
- `seereel-canvas-review` is the final consistency gate. It must compare the current canvas to the initial idea and locked `StoryPlan`; check node prompts, edges, `assetIds`, storyboard references, and continuity fields; then either PASS or route each failure back to `seereel-script-chat`, `seereel-casting-assets`, `seereel-cinematography`, `seereel-cli`, or `seereel-agent-session`.
- Storyboard references must be declared-grid cinematic contact sheets. A storyboard may use 2x2, 2x4, 3x3, or another justified X-grid, but the declared grid must be complete: every cell is present, each cell is a complete movie still, and panels are ordered as filmable story-action beats. Concept-board storyboards that only list themes, moods, symbols, or abstract ideas are not acceptable Seedance references; every panel must stage who does what, where, with what object/reaction/camera move, and how the shot state changes.
- The top bar credential entrypoint is named `API Keys`; inside it, users can save a BytePlus standard API key, a Volcengine CN standard API key, or an Agent Plan key.
- Seedream and Seedance standard API routes must use these endpoints unless explicitly overridden by route-specific env: BytePlus uses `https://ark.ap-southeast.bytepluses.com/api/v3`; Volcengine CN uses `https://ark.cn-beijing.volces.com/api/v3`; Agent Plan uses `https://ark.cn-beijing.volces.com/api/plan/v3`.
- Browser, CLI, and server-side Seedream/Seedance generation must resolve credential families in this order: BytePlus standard API, Volcengine CN standard API, Agent Plan. Within each family, request/browser credentials may be used before same-family environment credentials.
- The site/admin Agent Plan free-trial path is active only when no standard Ark API key is available for the request or environment and the user has not supplied their own Agent Plan key. A configured BytePlus or CN standard Ark key must not be blocked by Agent Plan free-trial user/IP/global quota.
- Session export packages must contain the visible session, shots, related assets, stitch jobs, prompt/review/render metadata, and local `/media/...` previews needed to reopen the workflow elsewhere.
- Session export packages must not contain API keys, Agent Plan keys, access tokens, admin credentials, cookies, or environment variables.
- Session import must create a new editable copy owned by the current browser identity. It must remap session, shot, asset, stitch, and render ids, and rewrite all internal references so the imported canvas keeps its graph links, first-frame/tail-frame continuity, reference-video wiring, stitch order, and local media previews.
- Importing a package must not overwrite an existing session unless the user explicitly chooses a future overwrite mode. The default UI path is copy-import.
- The session dock must support Shift-click range selection across the ordered session list. The user can delete the selected continuous range in one action.
- Deleting a session must also best-effort delete unshared intermediate artifacts referenced by that session: local `/media/...` previews, shot renders, stitch outputs, narration outputs, and TOS objects recorded as `tosObjectKey`. Cleanup must run after store deletion against the remaining snapshot and must preserve any local file or TOS object still referenced by another session, global asset, or Gallery item.
- Batch deleting sessions must use the same cleanup rule as single deletion, but collect candidates for all selected sessions before deleting any of them so local files or TOS objects shared only among the deleted range are cleaned, while objects still referenced by surviving sessions or Gallery items are preserved.

## Agent Skill System

- `.agents/skills/` is the single source of truth for project skills. Runtime-specific directories such as `.cursor/skills/`, `.claude/skills/`, `~/.codex/skills/`, `~/.claude/skills/`, `~/.cursor/skills/`, and `~/.agents/skills/` are install or mirror surfaces.
- `CLAUDE.md` must continue to point at `AGENTS.md`, so Claude Code reads the same project context as Codex and other agents.
- `npm run install:skill` must copy selected skills from `.agents/skills/` to detected or requested runtimes without making any runtime the canonical source.
- `packages/seereel-cli/skills/seereel-cli/SKILL.md` is the npm package's bundled CLI skill. It must stay in sync with `.agents/skills/seereel-cli/SKILL.md` whenever CLI behavior, credential guidance, handoff behavior, render behavior, or node-control behavior changes.
- `seereelcli skill install --agent all` installs only the bundled `seereel-cli` skill to global runtime skill folders. It does not replace the full repo skill set.
- Agent-facing docs must describe the same skill model: `AGENTS.md`, `README.md`, `README.zh-CN.md`, `.agents/skills/*/SKILL.md`, and the CLI bundled skill should not disagree about source-of-truth paths, credential precedence, handoff URLs, or production safety boundaries.
- Agents should route user intent to skills as follows:

| Skill | Primary use |
| --- | --- |
| `seereel-shortdrama` | End-to-end orchestration, stage routing, approval gates, fallback ownership, and transition into approved render/stitch. |
| `seereel-script-chat` | Initial idea or source material -> researched, reviewed, locked `StoryPlan`; no asset creation or final Seedance prompt ownership. |
| `seereel-casting-assets` | Locked `StoryPlan` -> approved character/scene/prop/style assets, style bible, coverage table, and `assetIds`; no plot/dialogue or final prompt ownership. |
| `seereel-cinematography` | Locked `StoryPlan` plus approved assets -> storyboard plan, shot nodes, camera grammar, continuity wiring, and `rawPrompt`/`prompt`; no script or asset approval ownership. |
| `seereel-canvas-review` | Final consistency review of initial idea, locked `StoryPlan`, node prompts, edges, `assetIds`, storyboard references, continuity, and fallback routing before video generation. |
| `seereel-agent-session` | REST-driven session, asset, storyboard, shot, render, poll, TOS publish, and stitch operations against visible SeeReel state. |
| `seereel-cli` | Local or published CLI transport, configuration, workflow creation, handoff, node control, publish-to-TOS, render polling, repair commands, download, and stitch. |
| `seereel-storyboard-imagegen` | Cinematic declared-grid storyboard contact-sheet and clean-keyframe prompting for one shot; no full Seedance prompt ownership. |
| `vibe-creating-prompt` | Video prompt polishing when the user's input is emotional, atmospheric, memory-like, or Vibe Creating oriented. |

- The listed project skills may call into the same SeeReel APIs or CLI, but the visible web app state remains the production handoff surface.
- Adding, deleting, renaming, or materially changing a skill is an agent-workflow change and must update this spec, the relevant README/AGENTS guidance, and any affected smoke tests in the same change.

## Acceptance Criteria

- [ ] A video node can persist prompt edits while queued or rendering.
- [ ] Video/Shot prompt edits auto-save from the Inspector, and the Inspector does not render a manual Save button for prompt persistence.
- [ ] The UI clearly distinguishes saved prompt from submitted prompt when they differ.
- [ ] Retry and regeneration use the current saved prompt unless the user explicitly selects an older render.
- [ ] A ready or errored shot with stale historical `generating` render rows is not treated as actively generating; it does not keep the UI poller alive and does not block `/api/shots/:id/generate` from creating a new render.
- [ ] Transient shot poll failures during an active generation are suppressed for the first two consecutive failures per shot, reset after a successful poll, and become visible if they keep repeating.
- [ ] Generated shot videos and stitched final videos publish their local cache to TOS/CDN when configured, persist playback/download URL metadata, and download routes may redirect to accelerated remote URLs while canvas playback routes prefer stable local cached media.
- [ ] Generated image assets, shot sketches, storyboards, and storyboard panels cache remote provider outputs to local `/media/...` before persisting canvas-visible image URLs, so the canvas remains viewable after remote signed URLs expire.
- [ ] Review toggle behavior is the same in local and production builds.
- [ ] Stitching only uses ready shots and records the selected playlist order in visible state.
- [ ] A stitch button/panel can select canvas video nodes without requiring noisy shot-to-stitch canvas edges. The panel defaults to video creation order, supports drag-in, drag-reorder, and remove, and persists the selected playlist before stitching.
- [ ] Manually moved canvas nodes persist their positions in the session and refresh back into the same layout.
- [ ] Right-click-created nodes and uploaded media nodes appear at the clicked/dropped canvas position without a fixed center offset.
- [ ] Shot and storyboard `@` mention menus list only graph-wired references, and server-side Seedream/Seedance reference resolution ignores unconnected `@` names even when stale prompt text still contains them.
- [ ] `npm run smoke:image-reference-binding` passes and proves Image-node Seedream prompts rewrite prompt-mentioned `@资产名` handles to `参考图 N`, do not append a reference-binding section, and otherwise avoid AI expansion.
- [ ] `npm run smoke:seedream-reference-payload` passes and proves Seedream reference-image requests use the official ordered `image` array, single-image output mode, 14-reference cap, and model-gated PNG output format.
- [ ] `npm run smoke:seedance-reference-binding` passes and proves Seedance text rewrites prompt-mentioned `@XXX` handles to Playground-style `【Pictures N】` / `【Video N】` / `【Audio N】` labels, binds them to the actual submitted `reference_image` / `reference_video` / `reference_audio` ordinal, and keeps non-`@` prose and stale/unsubmitted handles unbound.
- [ ] Connected references are resolved by asset id at generation time, so updating an asset changes what dependent nodes use on their next generation without reconnecting the edge.
- [ ] Updating an existing image asset's current media without an explicit thumbnail updates the stored thumbnail to the same current image, so canvas and Inspector previews cannot keep showing an older/other image.
- [ ] Image nodes default Seedream generation to 2K, Inspector offers 2K/4K resolution choices, and the selected value is saved on the asset and sent on generation.
- [ ] `npm run smoke:session-asset-scope` passes and proves script/storyboard planning does not import unrelated global assets into a new session, while preserving session-owned assets and explicitly wired global assets.
- [ ] Tail-frame extraction returns the source video's strict final decoded frame and preserves the decoded frame dimensions.
- [ ] Video nodes are labeled as video nodes and expose first-frame, tail-frame, and tail-video actions; each action creates a standalone canvas node without an automatic source edge from the video node, and the tail-video node stays movable from node chrome outside its player.
- [ ] Rendered video nodes and derived tail-clip video nodes expose an inline canvas player with native play/pause controls, a progress bar, and draggable seek without opening a separate lightbox.
- [ ] Rendered video nodes, imported/reference video nodes, tail-clip video nodes, and derived video processor nodes display their duration on the canvas header so stitch order and abnormal timelines are visible before final export.
- [ ] Newly created video/Shot nodes default to 15 seconds, and the Shot Inspector falls back to 15 seconds when a duration is omitted or cleared.
- [ ] Final-video stitching normalizes each selected video segment's timestamps before concat and rejects cached final videos that are far longer than the selected shot durations.
- [ ] Full-video stitch state marks an existing final video stale when selected source videos change, keeps the old final video playable, and surfaces newly ready videos outside a custom playlist without automatically inserting or restitching them.
- [ ] The Full video playlist shows an inline "已更新" badge beside selected videos whose current render is newer than the current final video.
- [ ] Shot-to-Shot video nodes can be connected before the source shot has a rendered video; the connection remains visible and persists as `referenceVideoFromShotId`.
- [ ] `npm run smoke:shot-tail-clip-asset` proves a rendered shot can create a session-scoped 2-second tail-clip asset that appears as a connectable reference-video node for the next shot.
- [ ] The Shot Inspector does not expose "参考上一镜尾段" / "Reference previous tail clip"; tail continuity is created with the visible tail-clip video action and explicit reference-video wiring.
- [ ] No generated video workflow emits subtitle files or burns subtitles by default; the post-stitch audio-track node exposes only "no subtitles" and "burn subtitles into video" modes, with a selectable burn-in position.
- [ ] A narration script with leading timecodes such as `[00:15]` starts those lines at the requested video timestamps while keeping `narrationSubtitleMode: none` free of subtitle streams.
- [ ] The post-stitch audio-track node can switch to music mode, submit a `GenBGMForTime`/`GenSongForTime` task through Volcengine OpenAPI AK/SK, poll `QuerySong`, persist the task id and audio URL in session state, and mix the returned music into the stitched video without using Agent Plan credentials.
